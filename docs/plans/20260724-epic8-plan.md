# Epic 8: Global UI/UX & Cross-Cutting Concerns Implementation Plan

## Goal Description
Complete Epic 8 for the BR Online Shop. Based on an initial review of the codebase, the Sticky Header, Footer, and Hamburger nav menu are already implemented in `Header.tsx` and `Footer.tsx`. We will focus on implementing global toast notifications, loading skeletons, error boundaries, rate limiting, fluent validation across all endpoints, API response standardization, accessibility passes, and SEO basics.

## Open Questions
- **API Response Wrapper**: Should the `ApiResponse<T>` wrapper be applied retroactively to all existing frontend fetching logic? This will require updating how the frontend extracts data from API responses.
- **Rate Limits**: Are there specific rate limit constraints you prefer (e.g., 5 requests per minute for login/contact forms)?

## Proposed Changes

### Frontend (`apps/web-shop`)
1. **Toast Notification System:**
   - Install `sonner` via shadcn/ui.
   - Update `layout.tsx` or `Providers.tsx` to include the `<Toaster />`.
   - Integrate toast notifications in cart actions, auth forms, and checkout flow.
2. **Global Loading States & Error Boundaries:**
   - Create `app/loading.tsx` with a global skeleton loader or spinner.
   - Create `app/error.tsx` for a global error fallback UI.
3. **SEO Basics:**
   - Update `app/layout.tsx` with proper standard Next.js Metadata (title, description, OpenGraph).
   - Add `app/sitemap.ts` and `app/robots.txt` for search engines.
4. **Responsive Breakpoints & Accessibility Pass:**
   - Manual audit and tweaks for mobile/tablet breakpoints across main pages.
   - Ensure `aria-labels` on all icon buttons and check contrast ratios.

### Backend (`apps/api-oos`)
1. **Rate Limiting:**
   - Register `Microsoft.AspNetCore.RateLimiting` in `Program.cs`.
   - Apply fixed window rate limiters to `/api/auth`, `/api/contact`, and `/api/applications` endpoints.
2. **API Response Standardization:**
   - Create an `ApiResponse<T>` wrapper class (e.g., `{ success: true, data: ..., message: ... }`).
   - Implement an `ActionFilter` or middleware to wrap all responses in a consistent JSON envelope.
3. **FluentValidation Coverage:**
   - Add validators for missing endpoints (`CartItemDto`, `CheckoutDto`, `ContactMessageDto`, `ApplicationDto`).
   - Ensure they are registered in the DI container.

## Verification Plan
1. **Manual Verification:**
   - Trigger a toast by adding an item to the cart or logging in.
   - Navigate routes to see `loading.tsx` state.
   - Throw a deliberate error to verify `error.tsx`.
   - Spam the login endpoint to verify HTTP 429 Too Many Requests response.
2. **Automated Validation:**
   - Send invalid payloads to the API to confirm `FluentValidation` properly rejects requests with structured error messages.
